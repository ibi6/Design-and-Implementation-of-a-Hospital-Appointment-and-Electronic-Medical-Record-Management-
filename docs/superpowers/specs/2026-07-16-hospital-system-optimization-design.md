# 慧医通全栈优化设计规格

**日期：** 2026-07-16
**状态：** 已批准连续执行（用户要求发现问题直接修正，不设置人工确认停顿）
**范围：** 产品、前端 UI/UX、后端、安全、数据一致性、性能、测试、部署与交付文档

## 1. 业务目标

### 1.1 用户与痛点

| 用户 | 核心场景 | 当前痛点 | 优化目标 |
|---|---|---|---|
| 患者 | 手机端找科室、选医生、预约、取消、查看病历 | 演示数据会过期；加载失败可能显示为空；部分触控目标不足 44px | 预约入口长期可用，异常可恢复，手机端操作清晰 |
| 医生 | 查看本人排班与预约、书写病历 | 病历属于敏感数据；接口错误状态不标准；缺少安全回归覆盖 | 严格按本人数据隔离，失败原因清楚，关键操作可审计验证 |
| 管理员 | 维护科室、医生、排班、用户并查看统计 | 手机表格横向滚动不直观；界面仍出现“Mock”旧文案 | 桌面高密度、手机卡片化，数据来源表述准确 |
| 开发/运维 | 本地运行、测试、容器部署、回滚 | Compose 只有 MySQL；默认密钥和 root 密码不安全；无前端测试 | 一套命令可运行，配置外置，构建测试可复现 |

### 1.2 商业与项目价值

- 保留“预约 → 接诊 → 病历 → 管理”完整闭环，提升答辩与演示可信度。
- 消除号源并发超卖、越权读取、Token 长期暴露等上线阻断风险。
- 让新人能从文档直接完成本地开发、测试、Docker 部署和回滚。
- 不引入支付、医保、住院、消息推送等范围外模块，避免过度设计。

### 1.3 核心流程

1. 患者登录后读取当前用户信息。
2. 患者浏览启用科室、医生及未来号源。
3. 数据库以原子更新预占号源，再创建预约；失败时事务回滚。
4. 医生只能读取本人预约，并为待就诊预约创建唯一病历。
5. 创建病历与预约完成处于同一事务。
6. 管理员维护基础资料与排班，并查看聚合统计。

## 2. 功能分解与数据流

### 2.1 一级至三级模块

| 一级模块 | 二级模块 | 三级能力 | 输入 | 输出 | 依赖 |
|---|---|---|---|---|---|
| 身份安全 | 登录会话 | 登录、当前用户、退出、失败限流 | 用户名、密码、Cookie/Bearer | HttpOnly Cookie、用户视图 | Spring Security、JWT、用户表 |
| 患者门户 | 预约挂号 | 科室、医生、号源、创建、取消 | 医生/排班选择、症状 | 预约视图、号源余量 | 排班、预约、事务 |
| 病历管理 | 接诊闭环 | 医生写病历、患者只读、权限隔离 | 预约 ID、结构化病历 | 病历视图、预约完成 | 预约、医生、病历表 |
| 管理门户 | 主数据 | 科室、医生、排班、用户状态 | 管理表单 | 更新后的实体视图 | RBAC、输入校验 |
| 数据看板 | 统计 | 用户、医生、科室、预约状态 | 当前日期 | 聚合统计 | 数据库聚合查询 |
| 体验系统 | 设计系统 | 导航、按钮、表单、弹窗、状态页 | 用户操作、加载结果 | 可访问反馈 | React、Tailwind |
| 交付运维 | 容器与代理 | 前端、API、MySQL、Nginx | 环境变量、镜像 | 可部署服务 | Docker Compose |

### 2.2 请求与数据流

```text
浏览器
  ↓ HTTPS / 同源 Cookie
Nginx（静态资源、/api 反向代理、安全响应头）
  ↓
Spring Security（认证、RBAC、限流、CORS）
  ↓
Controller（Bean Validation）
  ↓
Service（归属校验、事务、状态机）
  ↓
MyBatis-Plus / 参数化 SQL
  ↓
MySQL 8（生产）或 H2（本地演示）
```

Redis 不纳入当前范围：系统是单体、数据规模小，登录失败限流可先由进程内存实现；多实例部署时再将限流状态迁移到 Redis。

## 3. 技术方案

### 3.1 保留技术栈

- 前端：React 19、TypeScript 6、Vite 8、Tailwind CSS 4、React Router、Vitest、Testing Library。
- 后端：Java 21、Spring Boot 3.3、Spring Security、MyBatis-Plus、JWT、Bean Validation。
- 数据库：H2 用于零依赖演示，MySQL 8 用于生产。
- 部署：Docker 多阶段构建、Docker Compose、Nginx。

选择原因：现有架构边界清楚且构建通过，重写不会创造等量业务价值；应集中修复安全、并发、错误反馈和交付链路。

### 3.2 认证设计

- 登录成功同时写入 `HttpOnly; SameSite=Lax; Path=/` Cookie。
- 浏览器不再把 JWT 写入 localStorage；`fetch` 使用 `credentials: include`。
- 保留 `Authorization: Bearer` 读取能力，方便 Swagger、自动化测试和非浏览器客户端。
- 新增 `POST /api/auth/logout` 清除 Cookie。
- 生产环境通过 `APP_COOKIE_SECURE=true` 强制 Secure Cookie。
- 登录失败按规范化用户名计数；5 次失败后在 15 分钟窗口内返回 429，成功登录清零。

### 3.3 错误契约

响应体仍为 `{ code, message, data }`，HTTP 状态与错误码同步：

| 场景 | HTTP | code |
|---|---:|---:|
| 参数或业务状态错误 | 400 | 400 |
| 未认证/登录失败 | 401 | 401 |
| 无权限 | 403 | 403 |
| 资源不存在 | 404 | 404 |
| 登录尝试过多 | 429 | 429 |
| 未处理异常 | 500 | 500 |

服务端只记录内部异常，不把堆栈或数据库消息返回浏览器。

## 4. 数据库与一致性设计

### 4.1 关键约束

- `doctor.user_id` 唯一。
- `schedule(doctor_id, work_date, time_slot)` 唯一。
- 为预约的患者、医生、排班、状态以及病历患者、医生建立索引。
- 为核心关联建立外键，避免孤儿记录。
- `total_quota > 0`，`0 <= reserved_count <= total_quota`。

### 4.2 原子号源

预约创建先执行条件更新：

```sql
UPDATE schedule
SET reserved_count = reserved_count + 1, updated_at = CURRENT_TIMESTAMP
WHERE id = ?
  AND status = 'ACTIVE'
  AND reserved_count < total_quota;
```

受影响行数为 0 时返回“号源已满或停用”。预约插入失败时由事务回滚号源更新。取消使用下限保护的原子减法。

### 4.3 演示数据保鲜

初次启动创建角色、科室、医生和样例业务数据；每次启动都为启用医生补齐从今天起 7 天的上午/下午排班，已存在的排班不重复创建。这样演示库不会随时间永久失效。

## 5. API 设计变更

| 接口 | 方法 | 权限 | 变化 |
|---|---|---|---|
| `/api/auth/login` | POST | 公开 | 写 HttpOnly Cookie；失败限流；真实 HTTP 状态 |
| `/api/auth/logout` | POST | 可匿名 | 清除会话 Cookie |
| `/api/auth/me` | GET | 已认证 | Cookie 或 Bearer 均可 |
| `/api/appointments` | POST | PATIENT | 原子预占号源 |
| `/api/appointments/{id}/cancel` | POST | 归属用户/ADMIN | 原子释放号源 |
| `/api/stats/overview` | GET | ADMIN | 明确限制统计权限 |

既有业务成功响应结构不变，避免页面大面积重写。

## 6. UI / UX 设计

### 6.1 视觉方向

“克制的临床工作台”：

- 主色为深青绿，代表可信与医疗；天蓝用于信息、琥珀用于待办、玫红仅用于风险。
- 以清晰边界、轻微层次和局部渐变代替大面积玻璃态。
- 标题采用更有辨识度的中文字体栈，正文保持高可读性；不依赖外部字体网络。
- 页面内容宽度与密度按角色调整：患者强调行动入口，医生强调待办，管理员强调数据扫描效率。

### 6.2 交互与状态

- 所有按钮和导航触控高度不低于 44px。
- 移动抽屉和 Modal 使用 `role=dialog`、`aria-modal`、Esc 关闭、焦点进入/恢复、背景滚动锁定。
- 表单错误关联 `aria-describedby`，异步错误使用 `role=alert`。
- 列表统一提供加载、空、错误、重试四态；顶层错误边界处理未知渲染错误。
- 移除“Mock 实时统计”“数据保存在浏览器”等已失真的文案。
- 管理端用户表在手机端改为摘要卡片，桌面保留表格。

### 6.3 响应式

- 手机 375px：单列、44px 触控、管理数据卡片、抽屉导航。
- 平板 768px：双列统计与表单，导航仍可折叠。
- 桌面 1440px：固定侧栏、紧凑看板、多列信息布局。

## 7. 安全设计

- SQL 注入：只使用 MyBatis 参数绑定与 Lambda Query，不拼接用户 SQL。
- XSS：React 默认转义；不使用 `dangerouslySetInnerHTML`；配置 CSP。
- CSRF：浏览器 Cookie 使用 SameSite=Lax，同源 Nginx 代理；Bearer 客户端不依赖 Cookie。
- 权限绕过：服务层继续做患者/医生数据归属校验，管理接口用 `@PreAuthorize`。
- 暴力破解：用户名维度失败窗口限流，返回 429。
- Token 伪造：生产密钥必须从环境变量注入，至少 32 字节；生产禁用默认密钥。
- 敏感泄露：Cookie HttpOnly，服务端异常返回通用消息，日志不记录密码或 Token。
- 开发工具暴露：生产 profile 禁用 H2 Console 与 OpenAPI UI。
- 文件上传：当前系统无上传接口；未来新增时必须校验大小、MIME、扩展名并隔离存储。

## 8. 性能设计

- 统计由数据库 `COUNT`/条件查询完成，不把全表加载到 JVM。
- 号源扣减合并为单条条件更新，减少竞态与往返。
- 前端按路由懒加载，降低首次 JavaScript 体积。
- Nginx 对带内容哈希的静态资源设置长期缓存和 gzip。
- 当前数据量不引入 Redis；若多实例或热点查询增长，再缓存科室/医生目录并以管理更新主动失效。

## 9. 测试与验收

- 后端单元/集成：HTTP 状态、Cookie 登录/退出、暴力破解、RBAC、数据隔离、预约/病历闭环、号源边界。
- 前端单元：HTTP 客户端、认证路由、按钮与弹窗可访问性、错误状态。
- 构建：`npm run lint && npm test -- --run && npm run build`；`mvn test && mvn package`。
- 安全：`npm audit --omit=dev`、配置扫描、无 localStorage JWT、生产 profile 启动校验。
- 视觉：375、768、1440 三宽度检查首页、患者工作台、管理看板、管理列表和弹窗。
- 用户流程：患者预约 → 医生写病历 → 患者查看 → 管理员看板。

## 10. 部署、备份与回滚

- Compose 启动 MySQL、后端、前端 Nginx；健康检查控制依赖顺序。
- `.env.example` 列出 JWT、数据库密码、Cookie、CORS 和端口，不包含真实密钥。
- MySQL 每日 `mysqldump`，保留 7 日；恢复前先停止写入并校验备份。
- 镜像使用不可变版本标签；回滚时切回上一版本镜像并恢复兼容数据库备份。
- 本次不引入破坏性迁移工具；正式持续迭代应接入 Flyway。

## 11. 自检

- 无 TBD/TODO 或占位要求。
- 设计保留现有成功响应契约和三角色范围，与原规格不冲突。
- 安全、数据、UI、测试、部署均有明确实现与验收路径。
- 范围适合单个优化迭代；支付、Redis、多院区和消息系统明确排除。
